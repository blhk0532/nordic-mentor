Researchers Discover Critical GitHub - RCE Flaw Exploitable via Single Git Push

Published on Apr 30, 2024 by Bjorn Leon Henry

Cybersecurity researchers have disclosed details of a critical security vulnerability impacting GitHub.com and GitHub Enterprise Server that could allow an authenticated user to obtain remote code execution with a single "git push" command.

One git push Is All It Takes: Critical GitHub Flaw CVE-2026-3854 Enabled Remote Code Execution

A single git push command. That's all an attacker needed to hijack GitHub's internal systems and run malicious code on the platform's backend infrastructure.

Security researchers have uncovered a critical command injection vulnerability in GitHub—tracked as CVE-2026-3854 —that allowed attackers with repository push access to achieve remote code execution (RCE) on affected instances. The flaw affects multiple GitHub offerings, including:

  • GitHub Enterprise Cloud (standard)
  • GitHub Enterprise Cloud with Data Residency
  • GitHub Enterprise Cloud with Enterprise Managed Users
  • GitHub Enterprise Server

The vulnerability stems from an improper neutralization of special elements —in plain terms, a failure to sanitize user-supplied input before using it in internal system communications.

How the Attack Worked: A Metadata Delimiter Nightmare

When a user pushes code to GitHub, various internal services exchange metadata about the operation. This exchange relies on structured headers that use special delimiter characters to separate fields. The problem? Attackers could inject their own delimiter characters through carefully crafted git push option values.

Because GitHub did not properly sanitize those user-supplied values, a malicious actor could inject additional metadata fields into internal service headers. Downstream services, trusting the metadata as "internal," would then process the injected fields as legitimate commands—leading to remote code execution.

“During a git push operation, user-supplied push option values were not properly sanitized before being included in internal service headers. Because the internal header format used a delimiter character that could also appear in user input, an attacker could inject additional metadata fields through crafted push option values.”
— Official GitHub Advisory

From Injection to Full Compromise

The research team from Wiz (who reported the flaw on March 4, 2026) demonstrated two realistic attack scenarios:

On GitHub Enterprise Server

By injecting a modified rails_env value, attackers could force Git hooks to run in an unsafe mode, bypassing sandbox protections. Then, through path traversal techniques, they redirected the hook directory to execute arbitrary files. The final result? Command execution as the git service user—giving full filesystem access, control over internal configurations, and the ability to read any repository on the server.

On GitHub.com (Cloud)

Even though custom hooks are normally disabled on the multi-tenant cloud platform, the same injection technique allowed attackers to set an internal enterprise-mode flag via metadata. This enabled code execution on shared storage nodes, potentially exposing millions of repositories across different customers and environments.

Severity: Critical (High severity score) — Easy to exploit, massive potential impact.

Patch Released – But 88% of Instances Remain at Risk

GitHub acted quickly. The company fixed the vulnerability within two hours of responsible disclosure and released patched versions for Enterprise Server:

  • 3.14.24
  • 3.15.19
  • 3.16.15
  • 3.17.12
  • 3.18.6
  • 3.19.3

However, according to Wiz researchers, at the time of their public report 88% of GitHub Enterprise Server instances were still vulnerable —meaning the vast majority of self-hosted or enterprise customers had not yet applied the patch.

No evidence of in-the-wild exploitation or customer data compromise has been found. Still, the severity and ease of exploitation make this a critical update for any GitHub Enterprise user.

A New Era of Vulnerability Discovery: AI Meets Closed Source

Wiz researchers noted that this flaw was discovered in closed-source code using AI-assisted analysis —a sign of how vulnerability research is evolving. Complex logic flaws that might evade manual review can now be surfaced by AI tools, especially when analyzing proprietary internal protocols.

“The vulnerability chain highlights a pattern that extends well beyond GitHub. When multiple services written in different languages pass data through a shared internal protocol, the assumptions each service makes about that data become a critical attack surface. In this case, one service assumed push option values were safe to embed verbatim.”
— Wiz Research Team

What You Should Do Now

  • If you use GitHub Enterprise Server – Upgrade immediately to one of the patched versions listed above. Do not wait.
  • If you use GitHub.com (cloud) – No action is required; the fix has already been deployed. However, this incident underscores the importance of understanding shared responsibility models even on SaaS platforms.
  • For all developers – Review your CI/CD pipelines and repository access controls. While this specific flaw required push access, it's a powerful reminder that even trusted actions like git push can carry unexpected risks when internal protocols assume innocence.

Final Takeaway

A single command. A missing sanitization. A delimiter that became a weapon. CVE-2026-3854 is a textbook example of how trust boundaries inside complex systems can break in surprising ways. GitHub fixed it fast—but the real lesson is for every engineering team building distributed systems: never assume that internal metadata is safe just because it came from a "trusted" service.
